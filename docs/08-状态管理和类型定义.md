## 1. 状态管理架构

### 1.1 整体架构概述

项目采用 **轻量级 Store 模式**，类似于 Redux 的简化版本，但基于 React 的 `useSyncExternalStore` 构建。核心由以下部分组成：

| 文件 | 职责 |
|------|------|
| `src/state/store.ts` | 通用 Store 工厂 (createStore) |
| `src/state/AppStateStore.ts` | AppState 类型定义和默认值 |
| `src/state/AppState.tsx` | React Provider + Hooks (已编译，含 React Compiler) |
| `src/state/selectors.ts` | 选择器函数 |
| `src/state/onChangeAppState.ts` | 状态变更副作用处理器 |
| `src/state/teammateViewHelpers.ts` | 队友视图状态操作 |

### 1.2 状态存储机制

#### 1.2.1 Store 接口 (`src/state/store.ts`)

```typescript
type Listener = () => void
type OnChange<T> = (args: { newState: T; oldState: T }) => void

export type Store<T> = {
  getState: () => T
  setState: (updater: (prev: T) => T) => void
  subscribe: (listener: Listener) => () => void
}
```

- **getState()**: 返回当前状态
- **setState(updater)**: 接收一个 `(prev: T) => T` 的更新器函数，返回新状态
- **subscribe(listener)**: 注册监听器，返回取消订阅函数

**核心实现逻辑**:
```typescript
export function createStore<T>(initialState: T, onChange?: OnChange<T>): Store<T> {
  let state = initialState
  const listeners = new Set<Listener>()

  return {
    getState: () => state,
    setState: (updater) => {
      const prev = state
      const next = updater(prev)
      if (Object.is(next, prev)) return  // 引用相等则跳过
      state = next
      onChange?.({ newState: next, oldState: prev })
      for (const listener of listeners) listener()
    },
    subscribe: (listener) => {
      listeners.add(listener)
      return () => listeners.delete(listener)
    },
  }
}
```

**关键设计决策**:
1. **不可变更新**: 每次 setState 都创建新状态对象
2. **引用相等优化**: 如果 `Object.is(next, prev)` 为 true，跳过所有通知
3. **单一 onChange 回调**: 允许注册一个新旧状态对比的回调，用于副作用
4. **Set 存储监听器**: 支持多个监听器，自动去重

### 1.3 状态更新模式

#### 1.3.1 Reducer 风格的更新

状态更新采用函数式更新器模式，类似于 React 的 `setState(prev => ...)`：

```typescript
// 示例：进入队友视图
setAppState(prev => {
  const task = prev.tasks[taskId]
  const prevId = prev.viewingAgentTaskId
  // ... 计算逻辑
  return {
    ...prev,
    viewingAgentTaskId: taskId,
    viewSelectionMode: 'viewing-agent',
    tasks,
  }
})
```

#### 1.3.2 状态变更副作用 (`src/state/onChangeAppState.ts`)

`onChangeAppState` 函数是状态变更的**唯一副作用入口**，负责将状态变化同步到外部系统：

**监听的关键状态变更**:

| 状态字段 | 副作用行为 |
|----------|-----------|
| `toolPermissionContext.mode` | 通知 CCR 远程同步模式变化 + SDK 状态流 |
| `mainLoopModel` (变为 null) | 从 settings 中移除模型设置 |
| `mainLoopModel` (非 null) | 保存模型设置到 userSettings |
| `expandedView` | 持久化到 globalConfig (showExpandedTodos / showSpinnerTree) |
| `verbose` | 持久化到 globalConfig.verbose |
| `tungstenPanelVisible` (ant-only) | 持久化 tmux 面板可见性 |
| `settings` 变化 | 清除 API Key / AWS / GCP 凭据缓存，重新应用环境变量 |

**模式同步逻辑**（核心单点控制）:
```typescript
const prevMode = oldState.toolPermissionContext.mode
const newMode = newState.toolPermissionContext.mode
if (prevMode !== newMode) {
  const prevExternal = toExternalPermissionMode(prevMode)
  const newExternal = toExternalPermissionMode(newMode)
  if (prevExternal !== newExternal) {
    notifySessionMetadataChanged({
      permission_mode: newExternal,
      is_ultraplan_mode: isUltraplan,
    })
  }
  notifyPermissionModeChanged(newMode)
}
```

```typescript

export function useAppState<T>(selector: (state: AppState) => T): T

// 获取 setAppState 更新器，不订阅任何状态（稳定引用）
```

**性能优化要点**:
- 使用 `useSyncExternalStore` 实现选择器订阅
- 通过 `Object.is` 比较选择结果，仅在值变化时触发重渲染
- 选择器应返回已有子对象引用，而非创建新对象

### 1.5 状态持久化

状态持久化通过多个层级实现：

| 持久化目标 | 存储方式 | 触发条件 |
|-----------|---------|---------|
| 用户设置 | `settings.json` (CLAUDE_CODE_DIR) | `updateSettingsForSource('userSettings', ...)` |
| 项目设置 | `.claude/settings.json` | `updateSettingsForSource('projectSettings', ...)` |
| 全局配置 | `globalConfig` (JSON 文件) | `saveGlobalConfig()` |
| 会话元数据 | `notifySessionMetadataChanged()` | 权限模式、ultraplan 模式变化 |
| 转录消息 | Session storage (日志文件) | 每条消息追加到日志 |

## 7. 关键设计模式总结

### 7.1 不可变性保证

- **DeepImmutable<T>**: 递归地将所有属性标记为 readonly (函数类型除外)
- **Updater 模式**: `setState(prev => ({ ...prev, field: newValue }))`
- **引用相等检查**: `Object.is(next, prev)` 跳过不必要的通知

### 7.2 性能优化

| 技术 | 应用场景 |
|------|---------|
| `useSyncExternalStore` | 选择器订阅，仅在切片变化时重渲染 |
| `Object.is` 比较 | 跳过相等的状态更新 |
| Mutable ref (messagesRef, writtenPathsRef) | 推测执行中避免数组展开 |
| Lazy loading (command.load) | 延迟加载命令实现 |
| Message cap (TEAMMATE_MESSAGES_UI_CAP = 50) | 队友消息数组大小限制 |
| Feature flags (bun:bundle) | 死代码消除，减少包大小 |

### 7.3 类型安全

| 技术 | 说明 |
|------|------|
| Branded types (SessionId, AgentId) | 防止 ID 类型混用 |
| Discriminated unions (PluginError, PermissionDecisionReason) | 类型安全错误处理 |
| Zod schemas | 运行时验证 Hook 输出 |
| Compile-time assertions (IsEqual) | SDK 类型与 Zod schema 匹配验证 |
| Exact types (Protobuf 生成代码) | 防止多余字段 |

### 7.4 循环依赖避免

- **纯类型文件**: `src/types/permissions.ts` 仅包含类型定义，无运行时依赖
- **Lazy require**: 在函数体内延迟 require 打破循环
- **Inline type checks**: 避免导入循环的运行时类型检查
