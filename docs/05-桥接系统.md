# 桥接系统 (Bridge System)

## 1. 桥接系统架构

### 1.1 整体设计

桥接系统是 Claude Code 实现 **Remote Control** 功能的核心基础设施。它允许用户通过 claude.ai 网页端或移动端远程控制本地运行 Claude Code 进程。

```
┌─────────────────────┐        ┌──────────────────────┐        ┌──────────────────────┐
│   Web/Mobile Client │◄──────►│    Anthropic Server   │◄──────►│   Local Claude Code   │
│   (claude.ai)       │  HTTPS │   (CCR + Environments │  HTTP/ │   (Bridge Process)   │
│                     │   /WS  │    API + Session      │  WS/SSE│                       │
└─────────────────────┘        │    Ingress)           │        └──────────────────────┘
                               └──────────────────────┘                   │
                                                                          │ spawn
                                                                          ▼
                                                            ┌──────────────────────┐
                                                            │   Child CC Process    │
                                                            │   (SDK mode)          │
                                                            └──────────────────────┘
```

### 1.2 两种桥接模式

系统支持两种桥接实现路径:

| 特性 | v1 (Env-based) | v2 (Env-less) |
|------|---------------|---------------|
| 入口 | `initBridgeCore` (replBridge.ts) | `initEnvLessBridgeCore` (remoteBridgeCore.ts) |
| 门控 | `tengu_ccr_bridge` | `tengu_bridge_repl_v2` |
| 环境注册 | 需要 (register/poll/ack/heartbeat/deregister) | 不需要 |
| 会话创建 | `POST /v1/sessions` + `environment_id` | `POST /v1/code/sessions` (无 environment_id) |
| 传输层 | HybridTransport (WS读 + POST写) | SSETransport (读) + CCRClient (写) |
| 认证 | OAuth token | JWT (worker_jwt) |
| 会话ID前缀 | `session_*` (兼容层) | `cse_*` (基础设施层) |

### 1.3 远程/本地通信模型

**上行 (本地 -> 服务器):**
- v1: HTTP POST 到 Session-Ingress (`/v1/session_ingress/ws/{sessionId}`)
- v2: HTTP POST 到 CCR (`/v1/code/sessions/{id}/worker/events`)

**下行 (服务器 -> 本地):**
- v1: WebSocket 连接到 Session-Ingress
- v2: SSE (Server-Sent Events) 流 (`/v1/code/sessions/{id}/worker/events/stream`)

## 2. REPL 桥接

### 2.1 初始化流程

`initReplBridge()` 是 REPL 环境下的桥接入口,执行以下步骤:

```
1. 门控检查 (isBridgeEnabledBlocking)
   ├── GrowthBook flag: tengu_ccr_bridge
   └── claude.ai 订阅验证

2. OAuth 认证检查 (getBridgeAccessToken)
   ├── 检查是否有有效 token
   ├── 跨进程死亡 token 回退机制 (bridgeOauthDeadFailCount >= 3)
   ├── 主动刷新过期 token (checkAndRefreshOAuthTokenIfNeeded)
   └── 刷新失败后跳过

3. 组织策略检查 (allow_remote_control policy)

4. 获取 baseUrl (getBridgeBaseUrl)

5. 派生会话标题 (title derivation)
   ├── 优先级: initialName > /rename > 最后一条用户消息 > 自动生成的 slug
   ├── 第 1 条消息: deriveTitle 占位符 + 生成 Haiku 标题
   └── 第 3 条消息: 基于完整对话重新生成标题

6. 获取 orgUUID

7. 选择实现路径:
   ├── v2 (env-less): if tengu_bridge_repl_v2 enabled && !perpetual
   └── v1 (env-based): 默认路径

8. v1 路径:
   ├── 获取 git context (branch, gitRepoUrl)
   ├── 确定 workerType (claude_code 或 claude_code_assistant)
   └── 调用 initBridgeCore()
```

### 2.2 initBridgeCore 核心流程

```
1. 读取 bridgePointer (perpetual 模式下用于崩溃恢复)

2. 注册桥接环境 (registerBridgeEnvironment)
   └── POST /v1/environments/bridge

3. Perpetual 模式下尝试原地重连 (tryReconnectInPlace)
   └── POST /v1/environments/{id}/bridge/reconnect

4. 创建会话 (createSession)
   └── POST /v1/sessions (v1) 或 POST /v1/code/sessions (v2)

5. 写入崩溃恢复指针 (writeBridgePointer)

6. 启动工作轮询循环 (startWorkPollLoop)
   ├── 持续轮询工作项 (pollForWork)
   ├── 到达容量时发送心跳 (heartbeatWork)
   └── 收到工作时调用 onWorkReceived

7. 收到工作后:
   ├── 检查是否使用 CCR v2 (serverUseCcrV2 或 CLAUDE_BRIDGE_USE_CCR_V2)
   ├── v1: 创建 HybridTransport
   └── v2: 创建 SSETransport + CCRClient

8. 刷新历史消息 (initial flush)

9. 注册清理函数 (registerCleanup)

10. 返回 BridgeCoreHandle
```

### 2.3 工作轮询循环

轮询循环是 v1 桥接的核心调度机制:

```
while (!signal.aborted) {
  1. pollForWork(envId, envSecret)
     ├── 返回 null: 无工作
     └── 返回 WorkResponse: 有新工作

  2. 如果有工作:
     ├── acknowledgeWork(workId, sessionToken)
     ├── 调用 onWorkReceived(sessionId, ingressToken, workId, useCcrV2)
     └── 继续轮询 (transport 丢失时重连)

  3. 如果没有工作:
     ├── isAtCapacity() == true:
     │   ├── 发送心跳 (heartbeatWork)
     │   └── 在 at-capacity 间隔内等待
     └── isAtCapacity() == false:
         └── 在正常轮询间隔内等待
}
```

**心跳机制:**
- 心跳间隔默认 20s (服务器 TTL 60s, 3 倍余量)
- 带 ±10% 抖动以分散集群负载
- JWT 过期时 (401/403) 立即销毁 transport 并快速轮询重新分发

### 2.4 环境丢失恢复

当轮询返回 404 (环境被服务器回收) 时:

```
doReconnect():
  1. 关闭旧 transport, 捕获 SSE seq

  2. Strategy 1 - 原地重连:
     ├── 幂等重新注册环境 (registerBridgeEnvironment + reuseEnvironmentId)
     ├── 如果返回相同 env ID: reconnectSession()
     └── currentSessionId 保持不变

  3. Strategy 2 - 新会话:
     ├── 归档旧会话 (archiveSession)
     ├── 创建新会话 (createSession)
     ├── 更新崩溃恢复指针
     └── 清空 previouslyFlushedUUIDs
```

## 3. 远程桥接 (Env-less Bridge)

### 3.1 概述

`initEnvLessBridgeCore()` 实现了 v2 桥接路径,消除了 Environments API 层:

```
服务器 PR #292605 添加了 POST /bridge 端点,实现了 OAuth -> worker_jwt 的直接交换。
这使得 Environments API 层 (register/poll/ack/heartbeat/deregister) 对于 REPL 会话变为可选。
```

### 3.2 初始化流程

```
1. 获取 env-less 配置 (getEnvLessBridgeConfig)
   └── 从 GrowthBook flag tengu_bridge_repl_v2_config 读取

2. 创建会话 (createCodeSession)
   └── POST /v1/code/sessions (OAuth, 无 env_id)
       └── 返回 cse_* 格式的 session.id

3. 获取桥接凭证 (fetchRemoteCredentials)
   └── POST /v1/code/sessions/{id}/bridge (OAuth)
       └── 返回 { worker_jwt, expires_in, api_base_url, worker_epoch }

4. 创建 v2 传输 (createV2ReplTransport)
   ├── SSETransport (读取)
   └── CCRClient (写入 + 心跳)

5. 启动 JWT 刷新调度器 (createTokenRefreshScheduler)
   └── 在过期前 5 分钟主动刷新

6. 注册回调 (wireTransportCallbacks)
   ├── onConnect: 刷新历史消息
   ├── onData: 处理入站消息
   └── onClose: 401 恢复或其他失败

7. 连接传输

8. 注册清理函数
```

### 3.3 配置管理

`EnvLessBridgeConfig` 包含以下可配置参数 (通过 GrowthBook 远程配置):

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `init_retry_max_attempts` | 3 | 初始化阶段最大重试次数 |
| `init_retry_base_delay_ms` | 500 | 重试基础延迟 |
| `init_retry_jitter_fraction` | 0.25 | 重试抖动比例 |
| `http_timeout_ms` | 10000 | HTTP 请求超时 |
| `uuid_dedup_buffer_size` | 2000 | UUID 去重环形缓冲区大小 |
| `heartbeat_interval_ms` | 20000 | 心跳间隔 |
| `heartbeat_jitter_fraction` | 0.1 | 心跳抖动比例 |
| `token_refresh_buffer_ms` | 300000 | JWT 过期前刷新缓冲 (5 分钟) |
| `teardown_archive_timeout_ms` | 1500 | 清理时归档超时 |
| `connect_timeout_ms` | 15000 | 连接超时 |

## 4. 传输协议详解

### 4.1 传输抽象接口

`ReplBridgeTransport` 是 v1/v2 传输的统一抽象:

```typescript
type ReplBridgeTransport = {
  write(message: StdoutMessage): Promise<void>
  writeBatch(messages: StdoutMessage[]): Promise<void>
  close(): void
  isConnectedStatus(): boolean
  getStateLabel(): string
  setOnData(callback: (data: string) => void): void
  setOnClose(callback: (closeCode?: number) => void): void
  setOnConnect(callback: () => void): void
  connect(): void
  getLastSequenceNum(): number
  readonly droppedBatchCount: number
  reportState(state: SessionState): void
  reportMetadata(metadata: Record<string, unknown>): void
  reportDelivery(eventId: string, status: 'processing' | 'processed'): void
  flush(): Promise<void>
}
```

### 4.2 SSE 传输

`SSETransport` 实现了基于 HTTP 长连接的 SSE 读取:

**连接参数:**
- URL: `{baseUrl}/v1/code/sessions/{id}/worker/events/stream`
- 认证: `Authorization: Bearer {worker_jwt}`
- 支持 `from_sequence_num` 参数实现断点续传
- 支持 `Last-Event-ID` 头用于重连

**SSE 帧格式:**
```
event: stdout
id: 123
data: {"type":"assistant","message":...}

event: keepalive

```

**重连机制:**
- 指数退避: 基础 1s, 最大 30s
- 10 分钟重连预算 (RECONNECT_GIVE_UP_MS = 600,000ms)
- 保活超时 45s (服务器每 15s 发送一次 keepalive)
- 永久关闭码: 401, 403, 404

### 4.3 WebSocket 传输 (v1)

v1 使用 `HybridTransport`, 结合 WebSocket 读取和 HTTP POST 写入:

**WebSocket URL 构建:**
```typescript
// 生产环境
wss://{host}/v1/session_ingress/ws/{sessionId}

// 本地开发
ws://localhost:{port}/v2/session_ingress/ws/{sessionId}
```

**HTTP POST 写入:**
- 使用 `SerialBatchEventUploader` 进行批量上传
- 批量大小: 100 条
- 失败重试: 最多 50 次 (桥接模式)

### 4.4 CCRClient (v2 写入)

`CCRClient` 处理 v2 的写入路径:

**关键端点:**
| 方法 | 端点 | 用途 |
|------|------|------|
| PUT | `/worker/register` | 注册 worker, 获取 epoch |
| POST | `/worker/events` | 发送事件 |
| PUT | `/worker/state` | 更新 worker 状态 |
| PUT | `/worker/metadata` | 更新外部元数据 |
| POST | `/worker/events/{id}/delivery` | 确认事件交付 |

**Epoch 机制:**
- 每个 `/bridge` 调用递增服务器端 epoch
- 心跳/状态/事件请求必须携带当前 epoch
- Epoch 不匹配 (409) 时立即关闭传输

### 4.5 混合传输模式

v1 的 HybridTransport 组合了两种传输:
- **WebSocketTransport**: 处理入站事件 (从服务器接收)
- **SerialBatchEventUploader**: 处理出站事件 (POST 到服务器)

两者独立运行, POST 写入在 WebSocket 重连期间继续工作。

### 4.6 门控/缓冲机制 (FlushGate)

`FlushGate` 确保历史消息和新消息按正确顺序到达服务器:

```
状态机:
  start()   -> 进入门控状态, enqueue() 返回 true (消息入队)
  end()     -> 退出门控, 返回排队的消息用于排出
  drop()    -> 丢弃所有排队消息 (永久关闭时)
  deactivate() -> 清除门控标志但不丢弃消息 (传输替换时)
```

**使用场景:**
1. 初始连接时, 历史消息通过单次 HTTP POST 刷新
2. 刷新期间的新消息被排队
3. 刷新完成后, 排出排队消息

## 5. 认证和安全

### 5.1 JWT 处理

**WorkSecret 解码:**
```typescript
type WorkSecret = {
  version: number                // 必须为 1
  session_ingress_token: string  // JWT 或 OAuth token
  api_base_url: string
  sources: Array<...>
  auth: Array<{ type: string; token: string }>
  claude_code_args?: Record<string, string>
  mcp_config?: unknown
  environment_variables?: Record<string, string>
  use_code_sessions?: boolean    // 服务器驱动的 CCR v2 选择器
}
```

**JWT 刷新调度器:**
```
schedule(sessionId, token):
  1. 解码 JWT exp claim
  2. 计算延迟 = exp * 1000 - Date.now() - refreshBufferMs
  3. 如果延迟 <= 0: 立即刷新
  4. 否则: 设置定时器

doRefresh():
  1. 获取当前 OAuth token
  2. 调用 onRefresh(sessionId, oauthToken)
  3. 设置 30 分钟后续刷新定时器

v2 路径使用 scheduleFromExpiresIn():
  延迟 = expires_in * 1000 - refreshBufferMs (最小 30s)
```

### 5.2 信任设备

桥接会话在服务器端具有 `SecurityTier=ELEVATED`。信任设备机制:

```
门控: tengu_sessions_elevated_auth_enforcement (GrowthBook)

流程:
  1. /login 后: enrollTrustedDevice()
     └── POST /api/auth/trusted_devices
     └── 返回 device_token (90 天滚动过期)
     └── 存储到 keychain

  2. 桥接 API 调用:
     └── getTrustedDeviceToken()
     └── 添加 X-Trusted-Device-Token 头

  3. CLI 端和服务器端双重门控:
     └── CLI: 控制是否发送头
     └── 服务器: 控制是否验证头
```

### 5.3 工作密钥

**v1 认证:**
- 环境注册: OAuth token
- 工作轮询: environment_secret
- 会话入口: OAuth token (首选) 或 JWT

**v2 认证:**
- 会话创建: OAuth token
- 桥接凭证: `POST /bridge` -> worker_jwt
- 所有 CCR 请求: worker_jwt (包含 session_id claim 和 worker role)

### 5.4 权限回调

桥接通过 `sendPermissionResponseEvent()` 发送权限决策:

```typescript
type PermissionResponseEvent = {
  type: 'control_response'
  response: {
    subtype: 'success'
    request_id: string
    response: Record<string, unknown>  // { behavior: 'allow' } 等
  }
}
```

**端点:** `POST /v1/sessions/{id}/events`

## 6. 会话管理

### 6.1 会话 ID 兼容性

CCR v2 兼容层使用两种会话 ID 标签:

```
session_*  -> 客户端 API (兼容层, compat/convert.go:27 验证 TagSession)
cse_*      -> 基础设施层 (工作队列, CCR /worker/* 端点)

相同 UUID, 不同标签前缀。
```

**转换函数:**
```typescript
toCompatSessionId(id: string): string
  // cse_xxx -> session_xxx (当 cse shim 启用时)

toInfraSessionId(id: string): string
  // session_xxx -> cse_xxx

sameSessionId(a: string, b: string): boolean
  // 比较 UUID body, 忽略标签前缀
```

### 6.2 对等会话

`peerSessions.ts` 当前为存根实现 (`listPeerSessions() => []`)。

### 6.3 会话运行器 (SessionRunner)

`createSessionSpawner()` 在桥接多会话模式下生成子进程:

```
子进程参数:
  --print --sdk-url {url} --session-id {id}
  --input-format stream-json --output-format stream-json
  --replay-user-messages

环境变量:
  CLAUDE_CODE_OAUTH_TOKEN: undefined (剥离, 防止泄露)
  CLAUDE_CODE_ENVIRONMENT_KIND: 'bridge'
  CLAUDE_CODE_SESSION_ACCESS_TOKEN: {accessToken}
  CLAUDE_CODE_POST_FOR_SESSION_INGRESS_V2: '1'
  CLAUDE_CODE_USE_CCR_V2: '1' (v2 模式)
  CLAUDE_CODE_WORKER_EPOCH: {epoch} (v2 模式)
```

**子进程管理:**
- 解析子进程 stdout 的 NDJSON 输出
- 提取活动信息 (tool_use, text, result, error)
- 环形缓冲区: 最多 10 条活动, 10 行 stderr
- 权限请求转发: `can_use_tool` -> 服务器
- Token 刷新: 通过 stdin 发送 `update_environment_variables` 消息

## 7. Webhook 处理

### 7.1 Webhook 净化

`webhookSanitizer.ts` 当前为直通实现 (nop), 可能为未来功能预留。

### 7.2 入站附件处理

`inboundAttachments.ts` 处理来自 Web 端的文件附件:

```
流程:
  1. 从入站消息提取 file_attachments 字段
  2. 对每个附件:
     ├── GET /api/oauth/files/{uuid}/content
     ├── 写入 ~/.claude/uploads/{sessionId}/
     └── 返回 @path 引用
  3. 将 @path 引用前缀添加到消息内容

安全考虑:
  - 文件名消毒: basename + 替换非安全字符
  - UUID 前缀防止冲突
  - 最佳努力: 失败不影响消息传递
```

### 7.3 入站消息处理

`inboundMessages.ts` 处理入站用户消息:

- 支持字符串内容和 ContentBlockParam[] 数组 (包含图片)
- 规范化 image blocks: 修复 camelCase `mediaType` -> snake_case `media_type`
- 如果缺失 media_type, 通过图片数据自动检测格式

### 7.4 消息去重

`BoundedUUIDSet` 实现有界环形缓冲区去重:

```typescript
class BoundedUUIDSet {

  // 两个实例:
  //   recentPostedUUIDs: 出站消息回显过滤

}
```

## 8. 桥接 UI

### 8.1 状态机

桥接 UI 使用以下状态:

```
idle -> ready -> connected -> failed
            |
            +-> reconnecting -> connected / failed
```

### 8.2 显示元素

- **连接 URL**: `https://claude.ai/code/bridge?environment_id={id}`

### 8.3 远程模式 UI 适配

**出站仅模式 (outboundOnly):**
- 不打开 SSE 读取流
- 仅激活 CCRClient 写入路径
- 用于 CCR 镜像模式: 本地会话在 claude.ai 可见但不接收控制

- **redactSecrets**: 红化敏感字段 (session_ingress_token, environment_secret 等)
- **debugTruncate**: 截断长字符串 (2000 字符限制)

- **logBridgeSkip**: 集中化跳过日志 + 分析事件

## 10. 完整消息流

### 10.1 v1 消息流 (Env-based)

```
时间线:

[Web Client]                    [Server]                       [Local Bridge]
     |                              |                               |
     |-- 用户输入消息 ------------->|                               |
     |                              |-- 分配工作项 ---------------->| (pollForWork 返回)
     |                              |<-- acknowledgeWork -----------|
     |                              |                               |
     |                              |<-- 创建传输 (WebSocket) ------|
     |<-- 确认会话活跃 -------------|                               |
     |                              |                               |
     |                              |<-- 历史消息刷新 (POST) -------|
     |                              |                               |
     |-- 用户在 Web 打字 ---------->|                               |
     |                              |-- 通过 WebSocket 转发 ------->|
     |                              |<-- handleIngressMessage ------|
     |                              |   (回显过滤 + 路由)           |
     |                              |                               |
     |                              |<-- SDK 消息处理 --------------| (子进程执行)
     |                              |<-- 助手响应 (WebSocket) ----- |
     |<-- SSE 流式响应 -------------|                               |
     |                              |                               |
     |-- 工具执行需要权限 --------->|                               |
     |                              |-- control_request ----------->|
     |                              |<-- control_response ----------| (用户批准)
     |<-- 权限已解决 ---------------|                               |
```

## 11. 安全考虑

### 11.1 认证安全

- **OAuth Token**: 短期有效, 通过刷新机制续期
- **Environment Secret**: 环境注册时生成, 用于工作轮询认证
- **Worker JWT**: 包含 session_id claim 和 worker role, 有明确过期时间
- **Trusted Device Token**: 90 天滚动过期, 存储在系统 keychain

### 11.2 传输安全

- 所有通信使用 HTTPS/WSS
- JWT 不记录到日志 (debug 输出截断显示)
- 敏感字段在调试输出中自动红化

### 11.3 会话安全

- 会话 ID 验证防止跨会话污染 (sameSessionId 检查)
- UUID 去重防止消息重放攻击
- 环境隔离: 每个桥接实例有独立的 environment_id

### 11.4 访问控制

- 组织策略 `allow_remote_control` 控制远程访问权限
- 需要 claude.ai 订阅 (排除 Bedrock/Vertex 等部署)
- 完整 scope 的 OAuth token (排除 setup-token 等受限 token)
